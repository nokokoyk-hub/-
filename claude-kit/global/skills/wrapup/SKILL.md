# /wrapup — 作業終了の儀式（必ず完遂してから終わること）

セッションの記憶は消えても、ファイルに刻んだ記録は残る。次のちゃぴへの引き継ぎやで。
以下の 1〜6 を**すべて判定・実行してから**セッションを終えること。途中でやめない。

## 手順

0. **日時確認**：`TZ=Asia/Tokyo date` で日本時間を確認。以降の日付はすべて JST。

1. **今日の作業ダイジェスト作成**：`docs/digests/YYYY-MM-DD.md` を作成（同日2回目以降は追記）。内容：
   - 今日やったこと（変更したファイルと理由）
   - 決めたこと（設計判断・方針）
   - 見送ったこと・未解決のこと
   - 発生したエラーと解決方法（次回同じ穴に落ちないため）

2. **`docs/current_state.md`（北極星）の更新**：今日の変更を反映して、これ単体を読めば最新状態が分かる形に保つ。差分メモではなく、常に「現在の姿」を書く

3. **changelog 追記の判定**：リポジトリに changelog（changelog.html / CHANGELOG.md など）があれば、ユーザーに見える変更があったか判定し、あれば追記する。なければ「今日はユーザー向け変更なし」と判定だけ記録

4. **version 更新の判定**：バージョン番号を持つリポジトリ（package.json / 表示用 version など）なら、今日の変更が version 更新に値するか判定する。更新する場合は該当ファイルを更新。判定結果はダイジェストに書く

5. **学びの記録（`docs/lessons.md`）**：ちゃぴが経験から賢くなるための記録。以下の a〜d を順番にやる。

   a. **候補の洗い出し**：今日の作業・読んだファイル・参照した外部情報から、次回以降も役立つ学びを拾う
      - ❌ 事故：何が起きたか＋どう防ぐか（禁止ではなく「理由つきの原則」で書く）
      - ⭕ 成功：うまくいったパターンと、なぜうまくいったか
      - 💛 好み：のんの好み・進め方の希望
      - 外部情報（公式ドキュメント・調べもの等）由来の学びは、**出典と日付を必ず添える**（鮮度が分からんメモは事故のもと）

   b. **整合性チェック（関所）**：追記する前に、候補が以下と矛盾しないか必ず確認する
      - 共通 CLAUDE.md／リポジトリ直下の CLAUDE.md
      - `docs/ai_handoff.md`（触ってはいけない領域・開発ルール）
      - 既存の lessons.md の項目
      - 「推測」を「確定」のように書いていないか
      **矛盾があれば書き込まない**。対立点を整理してのんに提示し、どちらを正とするか裁定してもらう。ルールの衝突は人間が裁く

   c. **追記の実行**：チェックを通った候補を lessons.md に追記する（迷いがある候補だけ、のんに確認してから）。lessons.md がまだ無いリポジトリなら、下のテンプレートから作る

   d. **棚卸し**：既存の項目をざっと見て、古くなったもの・現状と食い違うものがあれば、更新・削除をのんに提案する。**メモは書くより捨てる方が大事**

6. **次回引き継ぎメモの作成**：`docs/handoff/YYYY-MM-DD.md` を作成。内容：
   - 次回やること（優先順）
   - 作業途中のもの（どこまで進んでいて、何が残っているか）
   - 注意点（触ってはいけない箇所・ハマりポイント）
   - **次セッション冒頭用の呼び出しコピー**：のんが次のセッションの最初に貼るだけで続きが始められる、短い指示文（例：「/kickoff して。前回は◯◯まで終わってて、今日は◯◯からやで」）

7. **コミット＆プッシュ**：上記のドキュメントとコード変更をすべてコミットしてプッシュする。のんにファイルを手動で上げさせない。コミットメッセージは簡潔な通常形式

8. **のんへの締めの報告**：
   - ✅ 今日できたこと
   - 📚 今日の学び（lessons.md に追記したもの・棚卸ししたもの）
   - 📦 コミット・プッシュしたもの
   - ➡️ 次回の最初の一歩
   - お疲れさまの一言（明るく！）

## lessons.md テンプレート（無いリポジトリに新規作成するときはこれを使う）

```markdown
# lessons.md — ちゃぴの経験メモ

> **これは地図であって檻やない。** 過去の経験則をまとめたもので、ちゃぴの行動範囲を狭めるためのものではない。
> 現状とメモが食い違うときは、盲目的に従わず「メモと状況が違うけどどうする？」とのんに確認すること。
> 各項目には日付・理由（出典）を必ず付ける。理由のない禁止は書かない。

## 🔒 絶対ルール（明示的に解除されるまで守る）

- （例）2026-07-08：認証・決済・DB スキーマは明示指示なしで触らない（ai_handoff.md 由来）

## 🧭 原則（基本従う。状況が変わっていたら再検討を提案）

- （例）❌ 2026-07-08：デプロイ前の環境変数確認を忘れてエラー → デプロイ前チェックリストに環境変数を入れる
- （例）⭕ 2026-07-08：300行超えたあたりでファイル分割したらバグが減った → 肥大化したら早めに分割を提案

## 💛 好み（のんの希望。柔らかく尊重）

- （例）2026-07-08：説明は短め＋例え話が好き
```

## 守ること

- 1〜6 は「作るか作らないか」を必ず判定すること。黙ってスキップしない
- lessons.md への追記は、**整合性チェック（5-b）を通す前に書き込まない**
- のんに渡すファイルが発生する場合は、差分ではなく完全なファイルで渡す
- 未コミットの変更を残したまま終わらない。コミットできない事情があるなら、それ自体を引き継ぎメモに書く
