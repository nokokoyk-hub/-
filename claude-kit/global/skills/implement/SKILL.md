# /implement — 実装パイプライン

まとまった実装を、どのモデルでも同じ品質で回すためのレールやで。以下の順番を崩さないこと。

## 手順

### 0. 正本確認
`docs/current_state.md`・`docs/ai_handoff.md` があれば読み、方針・禁止事項を確認する（このセッションで読み済みなら省略可）。

### 1. 事前整理（のんに提示して合意を取る）
以下の5点を整理して提示する：
1. 修正対象の機能
2. 関係する可能性があるファイル
3. 読むべきファイル
4. 変更が波及しそうな箇所
5. 変更しない方がよい箇所

### 2. サービス設定チェックリスト
コード以外に変更が必要な箇所を洗い出して提示する：
Supabase（テーブル・RLS・Edge Functions）／Vercel・Render（環境変数・設定）／外部 API（キー・Webhook）／認証設定 など。
「コード変更のみで完結」ならその旨を明言する。

### 3. 実装 — zundamon に依頼
Agent ツールで **zundamon** に依頼する。依頼文には必ず含めること：
- 実装内容と、手順1で整理した「読むべきファイル」「変更しない方がよい箇所」
- テスト・lint の実行方法（分かっていれば）

### 4. テスト・デバッグ — lamchan に依頼
zundamon の報告を受けたら、**lamchan** に依頼して検証する：
- 実装した機能が期待どおり動くか
- エッジケース・境界値・例外系の洗い出し
- 既存機能を壊していないか（波及確認）

lamchan がバグを見つけたら手順3に戻して zundamon に修正させる。**最大3往復**。3往復で解決しない場合はループを止めて、状況を整理してのんに相談する。

### 5. レビュー — reviewer に依頼
テストが通ったら **reviewer** にレビューを依頼する。
- 🔴 致命的の指摘 → 手順3に戻して修正し、再レビュー
- 🟡 懸念 → 対応するかのんに確認（軽微ならその場で修正してよい）
- 🟢 好み → 記録だけしてスキップ可

### 6. のんへの最終報告
- ✨ 何ができるようになったか（初心者にも分かる言葉で）
- 📁 変更したファイル
- ✅ テスト・レビューの結果
- ⚠️ 残っている懸念・手動でやってもらう必要があること（サービス設定など）

## 守ること

- 手順1・2 を飛ばしていきなり実装に入らない
- サブエージェントの報告を鵜呑みにせず、報告の「未確認・懸念」欄を必ず読んで対応を判断する
- 認証・決済・DB スキーマ・本番導線に触る必要が出たら、実装前にのんに確認する
- 軽い修正（数行・単一ファイル・波及なし）はこのパイプラインを使わず、ちゃぴ本体がその場で直してよい。ただし手順1の頭の中での確認は省略しない
